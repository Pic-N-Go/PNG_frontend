# Test Case Generator Skill

git diff + plan/spec을 기준으로 테스트 케이스 체크리스트를 생성하는 스킬입니다.

## 목적

- 변경사항(git diff + untracked)과 plan/spec을 기준으로 테스트 케이스를 체계적으로 생성한다.
- Unit / Integration / Manual 타입을 구분하고, 정상/경계/실패 시나리오를 각 1개 이상 포함한다.
- 자동 검증(tsc/lint)과 수동 확인 항목을 명확히 분리해 실행 가이드를 제공한다.

## 입력 우선순위

1. `docs/ai/plans/<feature-name>-plan.md` (Task 기준 — 있으면 반드시 참조)
2. `docs/ai/specs/<feature-name>.md` (AC 및 시나리오 기준)
3. `git diff HEAD` + untracked 파일 (실제 변경 범위)

plan이 없으면 git diff만으로 생성하되, 결과 첫 줄에 "**[주의] plan 연계 불가 — diff 기준으로만 케이스를 생성합니다**"를 표기한다. spec만 없는 경우는 경고 없이 plan + diff 기준으로 진행한다.

## 케이스 생성 규칙

1. **시나리오 타입 3종 필수**: 각 기능 단위마다 정상(Happy Path) / 경계(Edge) / 실패(Failure) 케이스를 최소 1개 이상 포함한다.
2. **테스트 타입 분류**:
   - `[Unit]` — 단일 함수·컴포넌트·훅의 독립 동작 검증 (외부 의존 없음)
   - `[Integration]` — API 연동, 상태 변화, 화면 전환 등 복수 레이어 연동 검증
   - `[Manual]` — 시뮬레이터/기기에서 직접 확인해야 하는 UI·UX 항목
3. **관련성 판단 (plan 없을 때)**: diff에 포함된 파일 경로와 변경 내용을 기준으로, 직접 수정된 함수·컴포넌트와 관련된 케이스만 생성한다. 전혀 관련 없는 케이스는 포함하지 않는다.
4. **수동 확인 항목 표기**: Manual 타입 케이스는 `TODO: <검증 항목>` 형식으로 표기한다.
5. **plan 범위 밖 케이스**: plan에 명시되지 않은 연관 기능(회귀 위험 등)은 "**권장 추가**" 섹션으로 분리한다. 본 체크리스트에 포함하려면 사용자 승인 후 추가한다.
6. **자동 검증 분리**: tsc/lint는 케이스 목록과 별도 섹션("자동 검증")에 표기한다. `--run-checks` 플래그 시에만 실제 실행하고 결과를 3) 섹션에 포함한다.
7. **중복 제거**: 동일 동작을 검증하는 케이스가 Unit과 Manual에 중복으로 나타나면 타입이 더 낮은 것(Unit)을 우선 유지하고 Manual을 생략한다.

## 코드 스니펫 추가 흐름

체크리스트 출력이 끝나면 "코드 스니펫도 추가할까요?" 사용자 확인을 요청한다.
- 승인 시: 각 `[Unit]` / `[Integration]` 케이스에 Jest 형태의 스니펫 예시를 추가한다.
  - 실제 실행 가능 코드가 아닌 구조 제안 형태로 제공한다 (테스트 프레임워크 미설정).
  - 파일명·함수명은 실제 변경 파일에서 가져온다.
- 거부 시: 체크리스트만 최종 출력으로 사용한다.

## 4-section 완료 보고

```
1) 대상 변경 요약
   - 변경/추가 파일 목록과 각 파일 1줄 요약

2) 생성된 테스트 케이스 목록
   ### 자동 검증
   - [ ] tsc: `pnpm exec tsc --noEmit`
   - [ ] lint: `pnpm lint`

   ### Unit
   - [ ] [정상] <케이스 설명>
   - [ ] [경계] <케이스 설명>
   - [ ] [실패] <케이스 설명>

   ### Integration
   - [ ] [정상] <케이스 설명>
   ...

   ### Manual
   - [ ] TODO: <검증 항목>
   ...

   ### 권장 추가 (plan 범위 밖 — 승인 필요)
   - <케이스 설명> — 이유: <회귀 위험 등>

3) 실행/검증 결과 (또는 실행 가이드)
   - --run-checks 미사용 시: "위 자동 검증 명령어를 직접 실행하세요."
   - --run-checks 사용 시: tsc/lint 실행 결과 상세

4) 남은 리스크
   - 미검증 항목, 테스트 프레임워크 미설정 상태, 회귀 가능성 등
```
